## Change Log — All Sessions

### Root Cause of Every Bug

This codebase was generated by 48 sequential AI orchestration prompts (`orchestration/prompt-01.md` through `prompt-48.md`). The code was never compiled or tested locally before being pushed to GitHub and deployed to Vercel. Every bug surfaced one at a time because the Vercel build pipeline stops at the first error. The pattern: fix one error → deploy → new error appears.

---

### Change 1 — Removed deprecated `export const config` from Stripe webhook route
**File:** `app/api/webhooks/stripe/route.ts`
**What:** Deleted lines 6–11:
```ts
export const config = {
  api: {
    bodyParser: false,
  },
}
```
**Why:** This is the Pages Router API config syntax. It is invalid in the App Router (`app/` directory). Next.js 14 throws a hard build error when it sees it. The route was already using `await req.text()` (correct App Router approach) so the config block was both wrong and redundant.

---

### Change 2 — Fixed wrong import paths in message route
**File:** `app/api/chat/[chatId]/message/route.ts`
**What:**
- Line 10: `import { callRise } from '@/lib/rise/openai-client'` → `import { callRise } from '@/lib/openai/client'`
- Line 11: `import { executeCompressionAsync } from '@/lib/rise/compression'` → `import { executeCompressionAsync } from '@/lib/memory/executor'`

**Why:** `lib/rise/openai-client.ts` and `lib/rise/compression.ts` do not exist. The AI that generated this route used the wrong paths. The actual files are `lib/openai/client.ts` (exports `callRise`) and `lib/memory/executor.ts` (exports `executeCompressionAsync`).

---

### Change 3 — Fixed non-existent export in memory trigger
**File:** `lib/memory/trigger.ts`
**What:**
- Line 1: `import { createServerClient } from '@/lib/supabase/server'` → `import { supabaseServer } from '@/lib/supabase/server'`
- Removed: `const supabase = await createServerClient();`
- All `supabase.from(` → `supabaseServer.from(`

**Why:** `lib/supabase/server.ts` exports a singleton `supabaseServer`, not a factory function `createServerClient`. The AI that generated this file confused this with Supabase's SSR client pattern. Since `supabaseServer` is already instantiated, calling it as a function throws at runtime.

---

### Change 4 — Fixed `recordRateLimitMessage` non-existent export
**File:** `app/api/chat/[chatId]/message/route.ts`
**What:**
- Line 9: `import { checkRateLimit, recordRateLimitMessage }` → `import { checkRateLimit, recordMessage }`
- Line 55: `const rateLimitAllowed = await checkRateLimit(userId)` → `const { allowed: rateLimitAllowed } = await checkRateLimit(userId)`
- Line 126: `await recordRateLimitMessage(userId)` → `await recordMessage(userId)`

**Why:** `lib/rise/rate-limit.ts` exports `checkRateLimit` and `recordMessage`. It does NOT export `recordRateLimitMessage`. Additionally, `checkRateLimit` returns `{ allowed: boolean, remaining: number }` — an object, not a boolean — so the original truthiness check `if (!rateLimitAllowed)` was always false (objects are truthy), meaning the rate limiter never blocked anyone.

---

### Change 5 — Replaced `jsonwebtoken` in middleware with Edge-compatible JWT
**File:** `middleware.ts`
**What:** Replaced `import jwt from 'jsonwebtoken'` and `jwt.verify(...)` with `import { verifySession } from '@/lib/auth/session'` and `await verifySession(...)`. Made the middleware function `async`.

**Why:** Next.js middleware runs in the Edge Runtime by default. `jsonwebtoken` uses Node.js-only APIs (`process.version`, `process.nextTick`) which are not available in Edge Runtime — this crashes at runtime for every authenticated request. The project already has a custom HS256 JWT implementation in `lib/auth/session.ts` that uses only `crypto.subtle`, which IS Edge-compatible. The `verifySession` function returns the same `{ user_id, subscription_status }` shape.

---

### Change 6 — Fixed TypeScript strict type error in session.ts
**File:** `lib/auth/session.ts`
**What:** Line 62: `crypto.subtle.verify("HMAC", cryptoKey, sigBytes, dataBytes)` → `crypto.subtle.verify("HMAC", cryptoKey, sigBytes.buffer as ArrayBuffer, dataBytes)`

**Why:** TypeScript 5.x tightened the generic type of `Uint8Array` to `Uint8Array<ArrayBufferLike>`. The `crypto.subtle.verify` signature expects `BufferSource` which maps to `ArrayBufferView<ArrayBuffer>` — a stricter type than `ArrayBufferView<ArrayBufferLike>`. The `.buffer as ArrayBuffer` cast is safe because `base64urlDecode` always creates fresh `Uint8Array` instances backed by plain `ArrayBuffer`, not `SharedArrayBuffer`.

---

### Change 7 — Upgraded Stripe SDK from v14.25.0 to v22.1.0
**File:** `package.json` (updated via `npm install stripe@latest`)
**What:** Stripe package version: `14.25.0` → `22.1.0`

**Why:** `lib/stripe/config.ts` specified `apiVersion: '2025-01-27.acacia'` which is newer than what v14 supports (v14 only supports up to `'2023-10-16'`). Upgrading to v22 — which actually supports `2026-04-22.dahlia` — was required.

---

### Change 8 — Updated Stripe API version to match SDK v22
**File:** `lib/stripe/config.ts`
**What:** `apiVersion: '2025-01-27.acacia'` → `apiVersion: '2026-04-22.dahlia'`

**Why:** Stripe SDK v22 requires its current API version (`2026-04-22.dahlia`). Specifying an older version string is a TypeScript type error in v22's strict type system.

---

### Change 9 — Fixed `current_period_end` on Subscription (Stripe breaking change)
**Files:** `app/api/webhooks/stripe/route.ts` and `lib/stripe/webhooks.ts`
**What:** Both files: `subscription.current_period_end` → `subscription.items.data[0].current_period_end` (replaced all occurrences with `replace_all: true` in each file)

**Why:** In Stripe API version >= 2024-09-30, `current_period_end` was removed from the top-level `Subscription` object and moved to each `SubscriptionItem`. All items in a subscription share the same billing cycle, so `items.data[0].current_period_end` is semantically equivalent to the old `subscription.current_period_end`.

## App Overview (context for vision generation)

**What RiseDial is:**
A subscription-based AI coaching PWA. Users sign up, select a monthly or annual plan, optionally add a "Premium Memory" add-on, and have ongoing coaching conversations with an AI (Rise) powered by OpenAI gpt-4o-mini. The app builds a psychological memory profile of each user over time using a memory compression system.

**Tech Stack:**
- Framework: Next.js 14 App Router (TypeScript)
- Database: Supabase (PostgreSQL, service role client, no RLS — all access via `supabaseServer`)
- Auth: Custom HS256 JWT via `crypto.subtle` — no third-party auth library
- Payments: Stripe (subscriptions + add-ons via webhooks)
- AI: OpenAI (gpt-4o-mini for chat, gpt-4o for premium memory compression)
- PWA: `next-pwa`
- Styling: Tailwind CSS
- Hosting: Vercel

**Key directories:**
- `app/` — Next.js App Router pages and API routes
- `lib/auth/` — JWT session management
- `lib/db/` — Supabase query functions (users, chats, messages, memory)
- `lib/rise/` — Core AI logic (system prompt, context window, API messages, rate limiting)
- `lib/openai/` — OpenAI client (`callRise`, `callCompression`)
- `lib/memory/` — Memory compression pipeline (trigger, compress, patch, executor)
- `lib/stripe/` — Stripe client, subscription logic, webhook handlers
- `lib/supabase/` — Supabase clients (server singleton, client-side anon)
- `lib/user/` — User initialization and onboarding
- `middleware.ts` — Edge-compatible JWT auth guard for API routes
- `orchestration/` — The 48 AI prompts that generated this codebase (context only)

**Database tables (from `supabase/migrations/001_initial_schema.sql`):**
- `users` — account, subscription status, Stripe IDs, plan type, premium memory flag
- `chats` — soft-deletable chat sessions per user
- `messages` — chat messages (user/assistant roles, user_message_index)
- `memory_profiles` — AI-generated psychological profiles (versioned JSON)
- `webhook_events` — idempotency log for Stripe webhooks
- `rate_limit_tracking` — 60-message / 60-minute rolling window per user

**Known gaps and risks (beyond the fixed build errors):**
1. No test suite exists — zero automated tests
2. Supabase schema may have drifted from what the code expects (no validation)
3. Stripe webhook handler logic exists in TWO places: `app/api/webhooks/stripe/route.ts` AND `lib/stripe/webhooks.ts` — the route file has its own inline handlers AND imports from the lib file (duplication risk, divergence risk)
4. The `PREMIUM_PRODUCT_ID = 'prod_URPiU0OHsZuXvk'` constant is defined in the webhook route but never used — dead code
5. Memory compression trigger checks message count at the `messages` table level but the compression executor operates at the chat level — potential mismatch
6. `lib/auth/subscription-gate.ts` exists but it's unclear if it's actually used by all routes that need it
7. Environment variable completeness is unverified — no startup validation beyond what Supabase/Stripe clients do
8. PWA configuration (`next-pwa`, `public/manifest.json`, `public/sw.js`) has never been tested
