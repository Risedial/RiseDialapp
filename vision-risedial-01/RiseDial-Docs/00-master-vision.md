## 1. PROJECT SUMMARY

### 1.1 What This Builds
A systematic audit, fix, and test pipeline that takes the RiseDial Next.js 14 App Router codebase from its current state — AI-generated, never locally validated, surfacing build errors one at a time in Vercel — to a fully passing build with zero TypeScript errors, all 14 structural concerns audited and corrected, and a complete Vitest + Playwright test suite covering 100% of `lib/` and all API routes.

### 1.2 The Problem It Solves
The RiseDial codebase was generated by 48 sequential AI orchestration prompts and pushed directly to GitHub without ever being compiled or tested locally. Every bug surfaces only when Vercel's build pipeline hits it — one at a time — making forward progress impossible to predict and shipping to real users impossible to trust. Developers cannot confidently deploy because no test suite, no type-clean build, and no verified end-to-end flow exists.

### 1.3 Who Uses It
The developer (Alex) who owns the RiseDial product. They use the pipeline output — a clean build, corrected code, and a passing test suite — to confidently deploy RiseDial and serve real paying users.

### 1.4 End-to-End Success Signal
The system is working when: `npx tsc --noEmit` exits 0, `next build` succeeds on Vercel, `npx vitest run` exits 0 with 100% line coverage of all files in `lib/` and `app/api/`, and `npx playwright test` exits 0 for all four E2E flows — all in a single CI run on GitHub Actions.

## 3. MODULE SPECIFICATIONS

### 3.1 Module 1 — Build Error Resolution

**Purpose:** Produce a clean `tsc --noEmit` and `next build` before any other module runs.

**Trigger:**
- Type: Manual
- Specifics: Developer runs `npx tsc --noEmit` and `next build` locally; failure output identifies what to fix

**Inputs:**
| Field | Type | Source | Required? | Constraints |
|-------|------|--------|-----------|-------------|
| Current TypeScript errors | text output | `npx tsc --noEmit` | Yes | Must list all errors — use `--noEmit` not build |
| `tsconfig.tsbuildinfo` | file | project root | Yes | Likely stale; file is in git untracked list |

**Outputs:**
| Field | Type | Destination | Format / Example |
|-------|------|-------------|-----------------|
| Zero-error tsc run | exit code 0 | terminal | `npx tsc --noEmit` exits with code 0, no output |
| Passing next build | Vercel build log | Vercel dashboard | Build completes without error |

**Step-by-step process:**
1. Delete `tsconfig.tsbuildinfo` from the project root. This file is a stale incremental build cache; its presence is the most likely cause of the phantom `Cannot find type definition file for 'testing-library__jest-dom'` error. TypeScript's `skipLibCheck: true` is already set in `tsconfig.json`, which should prevent this error unless the tsbuildinfo is pointing to a cached (now-removed) type entry.
2. Run `npx tsc --noEmit` and capture all remaining errors.
3. For each error, apply the minimal fix. Do not refactor surrounding code.
4. If the `@types/testing-library__jest-dom` error persists after deleting `tsconfig.tsbuildinfo`:
   - If: `node_modules/@types/testing-library__jest-dom` exists → the package was installed by a transitive dependency. Add `"types": ["node", "react", "react-dom"]` to `tsconfig.json` `compilerOptions` to restrict auto-discovery.
   - If: the package does not exist → run `npm install --save-dev @types/testing-library__jest-dom` to satisfy TypeScript's auto-discovery.
5. Run `npx tsc --noEmit` again. Repeat steps 3–4 until exit code is 0.
6. Run `next build` locally. Capture any additional build errors not caught by tsc (e.g., invalid App Router exports).
7. Fix any `next build` errors. The `export const config` Pages Router syntax error in `app/api/webhooks/stripe/route.ts` was already fixed in the prior session.
8. Commit when `tsc --noEmit` and `next build` both exit 0.

**Edge cases:**
| Scenario | System Behavior | User Notified? | Processing Continues? |
|----------|----------------|----------------|----------------------|
| Deleting tsbuildinfo causes new errors to appear | Those were pre-existing masked errors; fix them | No | Yes |
| `next build` fails with a different error than tsc | Separate category of error; investigate and fix independently | No | Yes |
| `jsonwebtoken` still imported in a file other than middleware | TypeScript error will surface; remove import and replace with `verifySession` from `lib/auth/session.ts` | No | Yes |

**Failure states:**
| Failure Type | Recovery Action | Data Impact |
|-------------|----------------|-------------|
| tsc errors cannot be resolved in isolation | Escalate to Module 2 audit; some errors may require understanding the full call chain | None |

**AI/LLM used:** No

**External integrations:** None

**Data stored:** None

### 3.2 Module 2 — Codebase Audit

**Purpose:** Perform a systematic read-only review of all 14 concern areas and produce a structured findings document before touching any additional code.

**Trigger:**
- Type: Module hand-off
- Specifics: Triggered after Module 1 (build passes). Read-only — no code changes during this module.

**Inputs:**
| Field | Type | Source | Required? | Constraints |
|-------|------|--------|-----------|-------------|
| All source files in `lib/`, `app/`, `middleware.ts` | TypeScript files | Local filesystem | Yes | |
| `supabase/migrations/001_initial_schema.sql` | SQL file | Local filesystem | Yes | Ground truth for schema |
| `package.json` | JSON | Local filesystem | Yes | |
| `tsconfig.json` | JSON | Local filesystem | Yes | |
| `public/manifest.json` | JSON | Local filesystem | Yes | PWA manifest |
| `public/sw.js` | JavaScript | Local filesystem | Yes | Service worker |

**Outputs:**
| Field | Type | Destination | Format / Example |
|-------|------|-------------|-----------------|
| `AUDIT.md` | Markdown file | Project root | Structured findings per concern area (see format below) |

**Audit areas — step-by-step:**

**Area 1 — Import and module resolution.**
1. For every `import` statement in `lib/`, `app/`, and `middleware.ts`: verify the imported path exists as a file and the named export exists with the exact name used.
2. Check specifically: `lib/openai/client.ts` exports `callRise` and `callCompression`; `lib/memory/executor.ts` exports `executeCompressionAsync`; `lib/supabase/server.ts` exports `supabaseServer` (singleton, not a function); `lib/rise/rate-limit.ts` exports `checkRateLimit` and `recordMessage`.
3. Flag any import that resolves to a non-existent path or non-existent named export.

**Area 2 — TypeScript strict type correctness.**
1. Verify `tsc --noEmit --strict` exits 0 (strict mode is already on in tsconfig.json).
2. Check Stripe SDK v22 type shapes: `Stripe.Invoice.customer` is `string | Stripe.Customer | Stripe.DeletedCustomer | null` — all casts must handle the full union.
3. Check `Uint8Array.buffer as ArrayBuffer` cast in `lib/auth/session.ts:62` is correct.
4. Check all `supabaseServer.from('table').select()` return types are properly narrowed.

**Area 3 — Runtime environment compatibility.**
1. List every file that `middleware.ts` imports (direct and transitive).
2. Verify none of those files import `bcryptjs`, `jsonwebtoken`, `fs`, `path`, `crypto` (Node built-in module, not Web Crypto API), or any package that requires Node.js runtime.
3. The `verifySession` function in `lib/auth/session.ts` uses only `crypto.subtle` — this is correct for Edge Runtime. Confirm no other Edge-imported file uses Node-only APIs.
4. All files under `app/api/` (except the webhook route, which already excludes the `runtime = 'edge'` declaration) run in Node.js runtime — confirm by checking for `export const runtime = 'edge'` declarations.
5. Check `lib/memory/executor.ts` line 1: `import 'server-only'` — this correctly prevents Edge Runtime use.

**Area 4 — Stripe integration correctness.**
1. Confirm Stripe SDK version is `^22.1.0` (confirmed in package.json).
2. Confirm `apiVersion` in `lib/stripe/config.ts` is `'2026-04-22.dahlia'`.
3. Audit ALL uses of `subscription.current_period_end` across the entire codebase — must be zero occurrences. All uses must be `subscription.items.data[0].current_period_end`.
4. **Webhook duplication finding:** `app/api/webhooks/stripe/route.ts` contains a complete inline implementation of all four event handlers (`checkout.session.completed`, `customer.subscription.updated`, `customer.subscription.deleted`, `invoice.payment_failed`) that duplicates `lib/stripe/webhooks.ts`. The route file does NOT import `routeWebhookEvent` from the lib file. The lib file's `routeWebhookEvent` export is dead code (never called). The route file has idempotency logic (webhook_events table check) that the lib file lacks. This duplication must be consolidated (see Module 3).
5. **Dead code finding:** `const PREMIUM_PRODUCT_ID = 'prod_URPiU0OHsZuXvk'` in the route file is declared but never referenced. Must be deleted.
6. Verify idempotency pattern: route file inserts into `webhook_events` BEFORE processing (crash-safe pre-insert). Confirm this pattern is preserved after consolidation.

**Area 5 — Supabase schema validation.**
1. For every `supabaseServer.from('table_name')` call across the codebase, verify `table_name` matches exactly one of: `users`, `chats`, `messages`, `memory_profiles`, `rate_limit_tracking`, `webhook_events`.
2. For every `.select('field')`, `.update({ field: value })`, and `.insert({ field: value })`: verify `field` exists as a column in the corresponding table per `001_initial_schema.sql`.
3. **RLS risk finding:** The migration enables RLS on all tables with policies using `auth.uid()` (Supabase's built-in auth system). The app uses a custom HS256 JWT and the `supabaseServer` service role client, which bypasses RLS entirely. The RLS policies are effectively dead code — they provide no protection and no access for browser-side queries. Document this in `AUDIT.md` as a security architecture note. Out of scope to fix; in scope to document.
4. Verify `memory_profiles` has a `UNIQUE` constraint on `user_id` (it does — confirmed in migration). This is required for the upsert pattern the compression code uses.

**Area 6 — Authentication flow.**
1. Trace the full sign-up flow: user submits email/password → password hashed with bcryptjs → user row inserted → `createSession(userId, subscriptionStatus)` called → cookie set via `setSessionCookie`.
2. Trace the full login flow: email/password submitted → `users` table query → bcryptjs compare → session created → cookie set.
3. Verify `middleware.ts` matcher covers all API routes that require authentication. Current matcher: `/api/chat/:path*`, `/api/memory/:path*`, `/api/subscription/:path*`, `/api/chats/:path*`, `/api/user/:path*`. Verify no authenticated route exists outside these paths.
4. **Security finding:** `lib/auth/session.ts` line 3: `const JWT_SECRET = process.env.JWT_SECRET ?? "changeme-insecure-fallback"`. If `JWT_SECRET` is not set in the environment, tokens are signed with a known public string. This is a critical security risk. Module 4 (env validation) must ensure `JWT_SECRET` throws on missing.
5. Verify `setSessionCookie` sets `httpOnly: true`, `sameSite: 'strict'`, `secure: true` — confirmed in current code.
6. Verify `verifySession` checks `exp` claim and returns `null` on expired tokens — confirmed.
7. Identify all API routes that read `x-user-id` and `x-subscription-status` headers set by middleware. Verify these routes do not have a fallback if the headers are absent (they shouldn't, since middleware blocks unauthenticated requests before they reach the route).

**Area 7 — Subscription flow.**
1. Trace the checkout creation flow: user selects plan → checkout session created with `metadata.user_id` → user redirected to Stripe.
2. Trace the webhook activation flow: Stripe fires `checkout.session.completed` → webhook route verifies signature → idempotency check → subscription retrieved with `expand: ['items.data.price']` → user record updated with `subscription_status: 'active'`, plan type, billing date, premium flag.
3. Verify `lib/auth/subscription-gate.ts` exists and is actually imported by every route that serves gated content. List any routes that serve premium content but do not call the subscription gate.
4. Verify the subscription gate reads `x-subscription-status` from the request header (set by middleware) and returns 402 or 403 for non-active subscriptions.

**Area 8 — Memory compression pipeline.**
1. Verify `user_message_index` is set on every user message insert. If messages are inserted without `user_message_index`, `checkCompressionTrigger` will never count them (it filters `.not('user_message_index', 'is', null)`), and compression will never fire.
2. Confirm trigger thresholds: initial compression fires at exactly 50 user messages in a chat; patch compression fires every 10 messages thereafter (at 60, 70, 80...).
3. Confirm `executeCompressionAsync` is called with `void` (fire-and-forget) in the message route so it does not block the HTTP response.
4. Verify `generateInitialProfile` and `patchMemoryProfile` both call OpenAI with JSON mode and parse the response correctly.
5. Verify `memory_profiles` upsert uses `user_id` as the conflict target (per the `UNIQUE` constraint).

**Area 9 — Rate limiting.**
1. **Race condition finding:** `recordMessage` reads the active window's `message_count`, increments it in application code, then writes back. Under concurrent requests (user sending messages quickly), two simultaneous requests can both read the same `message_count`, both compute the same incremented value, and both write it — effectively under-counting. Document this in `AUDIT.md`. Fix requires an atomic increment: either a Supabase RPC function or a raw SQL UPDATE with `message_count = message_count + 1`.
2. Verify the rolling window cutoff logic: `window_start >= now() - 60 minutes`. Confirm this is `gte` (greater-than-or-equal), not `gt`.
3. Verify `checkRateLimit` returns `{ allowed: boolean, remaining: number }` and the message route destructures it as `const { allowed: rateLimitAllowed }` (this fix was already applied).

**Area 10 — Dead code and duplication.**
1. Flag `const PREMIUM_PRODUCT_ID = 'prod_URPiU0OHsZuXvk'` in `app/api/webhooks/stripe/route.ts` — unused, must be deleted.
2. Flag `routeWebhookEvent` export in `lib/stripe/webhooks.ts` — never imported by the route. After webhook consolidation (Module 3), this becomes the primary export.
3. Check whether `jsonwebtoken` is still imported anywhere after the middleware fix. If not imported anywhere, remove it from `package.json` dependencies.
4. Check whether `@types/jsonwebtoken` in devDependencies can also be removed.
5. Identify any other files in `orchestration/` or elsewhere that are not part of the app runtime and should be excluded from TypeScript compilation via `tsconfig.json` `exclude` field if they cause type errors.

**Area 11 — Environment variable audit.**
1. Enumerate every `process.env.X` reference in the entire codebase. Expected complete list:
   - `SUPABASE_URL` — Supabase project URL
   - `SUPABASE_SERVICE_ROLE_KEY` — Supabase service role key
   - `JWT_SECRET` — HS256 signing secret (critical: must not have fallback)
   - `STRIPE_SECRET_KEY` — Stripe secret key
   - `STRIPE_WEBHOOK_SECRET` — Stripe webhook signing secret
   - `STRIPE_PRICE_MONTHLY` — Stripe Price ID for monthly base plan
   - `STRIPE_PRICE_ANNUAL` — Stripe Price ID for annual base plan
   - `STRIPE_PRICE_PREMIUM_MONTHLY_ADDON` — Stripe Price ID for premium memory monthly add-on
   - `STRIPE_PRICE_PREMIUM_ANNUAL_ADDON` — Stripe Price ID for premium memory annual add-on
   - `OPENAI_API_KEY` — OpenAI API key
   - `NEXT_PUBLIC_SUPABASE_URL` — Supabase URL for browser client
   - `NEXT_PUBLIC_SUPABASE_ANON_KEY` — Supabase anon key for browser client
   - `NEXT_PUBLIC_APP_URL` — App base URL (used in checkout redirects and password reset emails)
2. Verify this list is complete by grepping for `process.env.` across all files.
3. Flag any `process.env.X!` (non-null assertion without validation) — these will throw at runtime with an unhelpful error if the var is missing.

**Area 12 — PWA correctness.**
1. Verify `public/manifest.json` has required fields: `name`, `short_name`, `start_url`, `display`, `background_color`, `theme_color`, `icons` (with at least one 192×192 and one 512×512 PNG).
2. Verify the icon PNG files referenced in `manifest.json` exist in `public/`.
3. Verify `next-pwa` configuration in `next.config.js` (or `next.config.ts`): `dest: 'public'`, `register: true`, `skipWaiting: true`.
4. Verify `public/sw.js` is not empty and is a valid Workbox service worker (the file exists per git status).
5. Verify the app's root layout includes `<link rel="manifest" href="/manifest.json" />` and `<meta name="theme-color" ... />`.

**Purpose:** Apply every fix identified in `AUDIT.md` in dependency order, with one atomic git commit per fix.

| Consolidated webhook handler | TypeScript | `lib/stripe/webhooks.ts` | Single source of truth for all 4 event types + idempotency |

**Step-by-step process (webhook consolidation — the most complex fix):**
1. Move the idempotency check logic (webhook_events insert/lookup) from `app/api/webhooks/stripe/route.ts` into `lib/stripe/webhooks.ts` as a wrapper around `routeWebhookEvent`. The route file becomes a thin dispatcher: parse body, verify signature, call `lib/stripe/webhooks.ts` dispatcher.
2. Delete the inline handler functions (`getPlanTypeFromPriceId`, `detectPremiumItem`, `getBasePriceId`, `handleCheckoutSessionCompleted`, etc.) from the route file.
3. Delete `const PREMIUM_PRODUCT_ID = 'prod_URPiU0OHsZuXvk'`.
4. Verify the consolidated `lib/stripe/webhooks.ts` handles all 4 events, includes idempotency logic, and exports: `verifyWebhookSignature`, `routeWebhookEvent` (now includes idempotency).

**Step-by-step process (rate limiter atomicity fix):**
1. Replace the read-then-increment-then-write pattern in `recordMessage` with an atomic Supabase RPC call.
2. Create a Supabase database function `increment_message_count(p_user_id uuid)` that performs an atomic `UPDATE rate_limit_tracking SET message_count = message_count + 1 WHERE id = (SELECT id FROM rate_limit_tracking WHERE user_id = p_user_id AND window_start >= now() - interval '60 minutes' ORDER BY window_start DESC LIMIT 1)`. If no rows updated, insert a new window record.
3. Call this RPC from `recordMessage` via `supabaseServer.rpc('increment_message_count', { p_user_id: userId })`.

**Step-by-step process (JWT_SECRET security fix):**
1. In `lib/auth/session.ts`, remove the `?? "changeme-insecure-fallback"` fallback.
2. Change to: `const JWT_SECRET = process.env.JWT_SECRET` (bare, no fallback).
3. After Module 4 (env validation) adds `lib/env.ts`, import `JWT_SECRET` from there instead of reading `process.env` directly.

**Step-by-step process (all other fixes):**
- Each fix from AUDIT.md Areas 1–14 is applied as a separate commit with a message describing the area and the specific change.

### 3.6 Module 6 — Unit & Integration Test Suite

**Purpose:** Write Vitest tests covering 100% of lines, branches, and functions in `lib/` and `app/api/`.

**Trigger:**
- Type: Module hand-off
- Specifics: After Module 5 (test infrastructure passes empty run)

**Test isolation strategy:** `vi.mock()` at the module boundary. `supabaseServer` and the Stripe client are mocked in every test file that uses them. No real network calls. No real database.

### 3.7 Module 7 — E2E Test Suite

**Purpose:** Write Playwright tests covering the four critical user flows against a local `next dev` server.

**Trigger:**
- Type: Module hand-off
- Specifics: After Module 6 (unit/integration tests pass)

**Target environment:** `http://localhost:3000` via `npm run dev`

**E2E test isolation:** Tests use Stripe test mode. Tests use the separate `risedial-test` Supabase project. E2E tests make real HTTP calls to the local server which makes real Supabase and Stripe test calls.

**CI behavior:** The signup flow (Flow 1) requires Stripe webhook delivery which is not possible in standard CI. This flow is skipped in CI via the `SKIP_STRIPE_E2E=true` environment variable. Flows 2, 3, and 4 run in CI. Webhook handling is covered by unit tests in Module 6.

**Test user setup:** A `globalSetup.ts` script runs once before all tests. It uses the Supabase service role client directly to insert a test user row with `subscription_status='active'`. Credentials are stored in `e2e/fixtures/test-user.json`. `globalTeardown.ts` deletes the test user row (cascade-deletes all related data).
